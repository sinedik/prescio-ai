# prescio-ai — Roadmap

## Текущее состояние

### Архитектура
Event-driven микросервис: polling `analysis_queue` в Supabase → dispatch по типу → Claude API → запись результатов обратно в Supabase.

### Готово
| Job | Модель | Что делает |
|-----|--------|-----------|
| `analyze_market` | Sonnet 4.6 | Fair prob, edge_score, kelly, recommendation, thesis |
| `analyze_event` | Sonnet 4.6 + web search | Сводка события, ключевые факторы, сценарии, новости |
| `analyze_sport_odds` | Haiku 4.5 | Value bets по букмекерам, implied prob comparison |
| `match_cross_sources` | Haiku 4.5 | Матчинг спортивных событий с prediction markets |
| `check_news` | Haiku 4.5 + web search | Мониторинг новостей по событиям |
| `group_markets` | Haiku 4.5 | Группировка маркетов в реальные события |
| `ai_search` | Sonnet 4.6 + web search | Анализ пользовательских поисковых запросов |

### Промпты
Все промпты:
- Возвращают строго JSON
- Содержат математику (Kelly criterion, edge = fair_prob - market_prob)
- Запрещают Twitter/Reddit как источники
- Используют AP, Reuters, BBC, ISW как авторитетные источники

### Известные проблемы
| Проблема | Приоритет |
|----------|-----------|
| Нет мониторинга расходов — не знаем сколько тратим по типу запроса | Critical |
| Re-analyze одинаковых данных — нет дедупликации если маркет не изменился | High |
| Промпты не оптимизированы по токенам — есть дублирование в system prompts | High |
| Нет тестирования качества ответов — нельзя сравнить промпт A vs B | High |
| `check_news` срабатывает слишком часто на "неинтересные" новости | Medium |
| Нет логирования токенов per job — сложно считать cost per analysis | High |
| Timeout 30s на Claude — недостаточно для сложных запросов с web search | Medium |

## Roadmap

### Phase 0 — Observability (сейчас, критично)
**Цель:** знать сколько стоит каждый тип анализа и где теряем деньги

| Задача | Приоритет | Критерий |
|--------|-----------|----------|
| Логировать `input_tokens` + `output_tokens` на каждый Claude call | Critical | В логах видно cost per job type |
| Добавить `cost_usd` в `analysis_queue` после завершения | Critical | Можно посчитать расход за день/неделю |
| Дедупликация: не анализировать маркет если `last_analyzed < 1h` и цена не менялась | High | Расход на Anthropic падает на 30–50% |
| Увеличить timeout до 60s для Sonnet + web search | High | Нет обрывов на сложных запросах |

---

### Phase 1 — Prompt Optimization (1–2 недели)
**Цель:** лучше качество при меньших расходах

| Задача | Приоритет | Критерий |
|--------|-----------|----------|
| Аудит всех system prompts — убрать дублирование, сократить токены | High | System prompts < 500 tokens каждый |
| `analyze_market`: убрать лишние поля из ответа если они не используются в UI | High | Output tokens -20% |
| `check_news`: добавить relevance score — пропускать если score < 0.5 | High | Меньше ложных срабатываний |
| Тестовый фреймворк: golden set из 20 маркетов → сравнивать качество ответа при изменении промпта | High | Можно итерировать промпты безопасно |
| `analyze_event` — разбить на 2 шага: сначала веб-поиск, потом анализ (меньше токенов в контексте) | Medium | Качество не хуже, токены -15% |

---

### Phase 2 — Esports AI (3–5 недели)
**Цель:** AI аналитика специфичная для Dota/CS2, а не только prediction markets

| Задача | Приоритет | Критерий |
|--------|-----------|----------|
| Новый job: `analyze_dota_match` — прогноз исхода матча на основе статистики команд | High | Возвращает win_prob, key_factors, confidence |
| Новый job: `analyze_cs2_match` — аналог для CS2 с учётом карт | High | `/cs2/:matchId/analyze` работает |
| Esports промпт: учитывать форму команд, head-to-head, патч | High | Более релевантный анализ чем generic event |
| Матчинг: связывать Dota/CS2 матчи с prediction markets (Kalshi/Polymarket) | Medium | `match_cross_sources` расширить для esports |

---

### Phase 3 — Quality & Scale (1–3 месяц)
| Задача | Приоритет |
|--------|-----------|
| A/B тестирование промптов — автоматически выбирать лучший по user feedback | High |
| Батч обработка — группировать похожие запросы для снижения API overhead | Medium |
| Fine-tuning / few-shot примеры — добавить примеры хороших анализов в промпты | High |
| Кэш результатов — если запрашивают тот же маркет повторно, отдавать из кэша | Medium |
| Персонализация — учитывать интересы пользователя при приоритизации очереди | Low |
